06

track in @for Performance

Without a proper track expression, Angular cannot identify which items changed and re-processes all N views on every list mutation. With track item.id, only genuinely new or updated items trigger DOM mutations.

✗ track $index HTML 

    


    

      

        ✓ track item.id
        HTML
      

      

    

  


  

    

      Live demo — watch the view re-process counter
      

        
      

    


    


      

        

          track $index
          

            
            
          

        

        

          DOM views processed
          {{ badOps() }}
        

        

          @for (item of badItems(); track $index) {
            

              {{ item.name }}
              
                ×{{ item.renders }}
              
            

          }
        

      


      

        

          track item.id
          

            
            
          

        

        

          DOM views processed
          {{ goodOps() }}
        

        

          @for (item of goodItems(); track item.id) {
            

              {{ item.name }}
              
                ×{{ item.renders }}
              
            

          }
        

      


    


    

      ×N shows how many times each view was re-processed by Angular.
      On the track $index side, all items increment on every operation — even unchanged ones.
      On the track item.id side, only the new item starts at ×1; existing items stay untouched.
    

  


  

    
💡

    

      When track $index is acceptable
      

        If the list is order-stable (never reordered, items only appended at the end,
        never removed from the middle) and items carry no internal DOM state (inputs, animations),
        $index is fine. In any other scenario — shuffle, prepend, arbitrary removal,
        pagination — prefer track item.id or another stable identity property.
